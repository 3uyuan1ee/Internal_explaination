# Concept Bottleneck Model 概念干预实验设计文档

## 1 研究问题

### 1.1 核心问题

Concept Bottleneck Model (CBM) 的设计初衷是让模型决策"可干预"——当模型预测错误时，领域专家可以通过修正中间的概念表示来纠正最终分类。然而，一个关键的现实问题始终悬而未决：

**在有限的专家认知预算下，应该如何选择需要干预的概念，才能最高效地纠正模型错误？**

原论文 (Koh et al., ICML 2020) 仅验证了随机干预和基于熵的分组干预的有效性，但未系统性地回答"干预策略的差异究竟有多大影响"。本实验从这一问题出发，围绕"干预效率"展开四个递进式子实验。

### 1.2 研究动机

CBM 的理论优势在于引入了人类可理解的语义概念层，使模型的决策路径可被审查和修正。但在实际部署中，专家的注意力和时间都是稀缺资源。如果需要修正大量概念才能纠正一个错误，或者不知道该优先修正哪些概念，那么 CBM 的"可干预性"就只是理论上的承诺，而非实际价值。

因此，本实验的核心目标不是复现 CBM 的分类性能，而是**量化分析干预行为本身**——错误从何而来、不同干预策略的效率差异、最少需要干预多少概念、以及干预在噪声条件下的鲁棒性。

## 2 模型设计

### 2.1 整体架构

```
Image → [ConceptPredictor] → ĉ (概念概率) → [LabelPredictor] → ŷ (类别 logits)
         InceptionV3 backbone    136 维          Linear + L1
         (partial freeze)     sigmoid 输出     权重矩阵可解释
```

CBM 将分类拆成两个阶段。ConceptPredictor 负责从图像中提取语义概念（如"有红色翅膀"、"有短喙"），LabelPredictor 基于这些概念做出分类决策。瓶颈结构的关键约束在于：分类器**只能通过概念层感知输入图像**，无法绕过概念进行"偷看"。

### 2.2 ConceptPredictor：InceptionV3 部分冻结

**架构选择**：采用 InceptionV3（ImageNet 预训练）作为视觉编码器，冻结至 Mixed_6e 层（约 64.7% 参数冻结），仅训练高层特征和概念预测头。

**设计理由**：

1. **为什么用 InceptionV3 而非 ResNet-18**：原论文使用 ResNet-18 作为 backbone，但 InceptionV3 在细粒度视觉任务上有更强的表征能力。CUB 数据集的类间差异极小（不同鸟种可能仅在喙形、翅膀条纹等细微处有区别），需要更强的特征提取能力来准确预测底层概念。

2. **为什么部分冻结而非全量微调**：底层特征（边缘、纹理、颜色）已经在 ImageNet 上充分学习，与鸟类的视觉概念高度相关。冻结底层可以：(a) 防止在小数据集上过拟合；(b) 大幅减少可训练参数（16.4M / 25.4M），降低训练成本；(c) 加速收敛。Mixed_6e 是一个自然的分界点，其后的 Mixed_7 层负责更高语义的组合，适合针对鸟类概念进行适配。

3. **概念预测头的设计**：从 2048 维特征直接映射到概念 logits，使用 sigmoid 输出概率。每个概念独立预测（多标签分类），对应 CUB 数据集中的二元属性标注（如"有黑色翅膀"= 1 或 0）。

**损失函数**：BCEWithLogitsLoss，并使用 pos_weight 解决属性不平衡问题。CUB 数据集中，某些属性（如"有翅膀"）几乎为全 1，而另一些（如"有红色冠羽"）极为稀少。pos_weight = n_neg / n_pos 为每个属性赋予与频率成反比的权重，防止模型通过全部预测多数类来"偷懒"。

### 2.3 LabelPredictor：线性 + L1 稀疏

**架构选择**：纯线性层 `Linear(num_concepts, num_classes)`，无隐藏层。

**设计理由**：

1. **为什么是线性的**：这是整个实验可解释性的根基。线性映射意味着权重矩阵 $W \in \mathbb{R}^{K \times C}$（K 个类别，C 个概念）的每个元素 $W_{k,c}$ 直接表示"概念 c 对类别 k 的贡献大小"。如果引入非线性隐藏层，这种直接对应关系就会丧失——你无法再通过权重矩阵回答"这个分类决策依赖了哪些概念"。

2. **为什么用 L1 正则化**：$\mathcal{L} = \text{CrossEntropy} + \lambda \sum|W|$，$\lambda = 0.0001$。L1 正则化驱动权重向稀疏方向发展，使得每个类别的决策只依赖少量关键概念。这在干预场景中至关重要：如果分类依赖的概念越少，专家需要干预的概念也越少，干预效率自然越高。

3. **为什么用预测概念而非真实概念训练**：这是 CBM 的核心设计。训练 LabelPredictor 时使用 ConceptPredictor 的预测输出 $\hat{c}$ 而非真实标注 $c^{*}$。这样做的目的是模拟部署时的真实情况——在线推理时不可能获取真实概念标注，只有 ConceptPredictor 的预测可用。如果用 $c^{*}$ 训练，LabelPredictor 在推理时看到的 $\hat{c}$（包含预测噪声）会产生分布偏移。

### 2.4 训练流程

```
Stage 1: 训练 ConceptPredictor
          SGD(lr=0.01, momentum=0.9, wd=4e-5)
          StepLR(step=300, gamma=0.1)
          EarlyStopping(patience=100)

Stage 2: 训练 LabelPredictor（冻结 ConceptPredictor）
          SGD(lr=0.001, wd=5e-5)
          StepLR(step=500, gamma=0.1)
          CrossEntropy + L1(lambda=0.0001)
          EarlyStopping(patience=100)
```

两个阶段顺序执行，Stage 2 依赖 Stage 1 的输出。这种"先学概念，再学标签"的分离式训练确保了概念层作为独立的信息瓶颈存在——标签预测器无法跳过概念层直接从图像特征中获取信息。

## 3 数据处理

### 3.1 类别选择

CUB-200-2011 包含 200 种鸟类，全部使用会导致类别过于密集、训练时间过长。我们使用固定随机种子（seed=42）从中无放回抽样 50 种，确保实验可复现。

50 类的选择在覆盖度和可分性之间取得了平衡：足够多以保持分类任务的挑战性，但又不过于密集以至于类间差异不可学习。

### 3.2 概念筛选

CUB 原始标注包含 312 个二元属性（如"有红色翅膀"、"有长喙"），但并非所有属性对分类都有贡献。我们通过**方差过滤**去除信息量不足的属性：

- 计算每个属性在训练集上的样本方差
- 剔除方差 $< 0.05$ 的属性（即在绝大多数样本中取值相同或缺失的属性）
- 保留约 136 个有区分力的属性

此外，原始属性标注包含置信度评分（1-4），我们仅使用置信度 $\geq 3$ 的标注，丢弃不确定的标注（统一设为 0）。

### 3.3 数据增强

训练阶段：RandomCrop、RandomHorizontalFlip、ColorJitter（亮度/对比度/饱和度各 0.2）。这些增强帮助模型学习对位置、朝向、光照变化鲁棒的概念表示。测试阶段仅使用 CenterCrop，确保评估的一致性。

## 4 实验设计

四个实验从宏观到微观、从理想到现实，形成一条完整的分析链：

```
实验 1: 错误归因 ──→ 错误从哪里来？
实验 2: 策略比较 ──→ 理想条件下不同策略的差异有多大？
实验 3: 最小干预 ──→ 每个样本最少需要干预多少？
实验 4: 噪声鲁棒 ──→ 现实条件（不完美的专家）下干预还有效吗？
```

### 4.1 实验 1：错误归因

**目标**：将 CBM 的分类错误分解为两个来源——概念预测器错误和标签预测器错误。

**思路**：CBM 的决策链是 $\text{Image} \xrightarrow{f_\theta} \hat{c} \xrightarrow{g_\phi} \hat{y}$。对于每一个被错分的样本，我们可以构造一个"反事实"问题：**如果概念预测是完美的（使用真实概念 $c^*$ 代替 $\hat{c}$），分类是否仍然错误？**

- 如果 $g(c^*) = y^*$ 正确但 $g(\hat{c}) \neq y^*$：错误归因于概念预测器（concept-attributable error）。说明 ConceptPredictor 没有正确预测某些关键概念，导致下游分类出错。这类错误**可以通过概念干预来修正**。
- 如果 $g(c^*) \neq y^*$ 也错误：错误归因于标签预测器（label predictor error）。说明即使概念完美，线性映射也无法正确分类——可能是该类别的决策边界本质上是高度非线性的。这类错误**无法通过概念干预来修正**。

**为什么重要**：这个分解直接决定了干预的"天花板"。如果绝大多数错误都是概念层面的（这通常是 CBM 的预期情况），那么干预策略有巨大的发挥空间。反之，如果标签预测器本身就有大量错误，那么再精巧的干预策略也无法解决问题——这时候应该回头改进模型架构。

**附加分析**：对于概念归因错误，进一步统计每个错误概念的贡献权重 $\sum_{\text{error samples}} |W[y^*, c_i]|$，识别出哪些概念的预测失败对最终分类影响最大。

### 4.2 实验 2：策略比较

**目标**：在控制干预预算 $k$ 的条件下，比较五种干预策略的效率。

**核心变量**：干预概念数 $k \in \{0, 1, 2, 3, 5, 8, 10, 15, 20, 30, 50, \text{all}\}$。$k=0$ 即无干预（CBM 基线），$k=\text{all}$ 即全部概念用真实值替换（Oracle 上界）。

**五种策略的设计思路**：

| 策略 | 选择依据 | 现实场景类比 |
|------|----------|-------------|
| **Random** | 随机选择 k 个概念 | 盲目干预，作为基线 |
| **Uncertainty** | 选择 $|\hat{c}_i - 0.5|$ 最小的 k 个（模型最不确定的概念） | 专家优先检查模型"犹豫不决"的属性 |
| **Importance** | 选择 $|W[\hat{y}, i] \cdot \hat{c}_i|$ 最大的 k 个（对当前预测影响最大的概念） | 专家优先检查对当前分类决策权重最大的属性 |
| **Greedy Oracle** | 贪心地逐个选择使 $P(y^*)$ 提升最大的概念（需要知道真实标签） | 理想最优策略，但需要知道答案才能使用 |
| **Error-Targeted** | 仅修正预测错误的概念，按 $|W[y^*, i]|$ 排序 | 专家发现模型某些属性预测错误，优先修正对真实类别影响最大的 |

**策略设计的递进关系**：

- **Random** → 最朴素的基线，不利用任何信息
- **Uncertainty** → 利用了模型自身的置信度信息，不依赖外部知识
- **Importance** → 利用了标签预测器的权重结构，关注"模型认为重要"的概念
- **Error-Targeted** → 利用了错误定位信息（哪些概念预测错了），是最具实践意义的策略
- **Greedy Oracle** → 利用了完美信息（知道真实标签和每个概念的真实值），是理论上界

这五种策略从"零信息"到"全信息"构成了一个完整的谱系。Uncertainty 和 Importance 是**部署时可用**的策略（不需要知道真实标签），而 Error-Targeted 和 Greedy Oracle 代表了**不同程度的理想假设**。

**评估指标**：AUC（准确率-干预数量曲线下的面积）。AUC 越大，说明该策略在各个预算水平下整体表现越好。这比单一 $k$ 值的比较更全面。

### 4.3 实验 3：最小充分干预

**目标**：对每个被错分的样本，找到能将其纠正所需的最少概念数 $k_{\min}$。

**方法**：贪心搜索。对每个错误样本：
1. 从 $k=0$ 开始（无干预）
2. 尝试修正每一个未修正的概念，选择使 $P(y^*)$ 提升最大的一个
3. 如果预测已正确，记录当前 $k$ 值；否则继续

**为什么用贪心而非全局最优**：全局最优需要枚举 $\binom{C}{k}$ 种组合（$C \approx 136$），计算代价极高。贪心搜索的时间复杂度为 $O(k_{\min} \cdot C)$，对于大多数样本在实践中是可接受的。此外，由于 LabelPredictor 是线性的，贪心策略在大多数情况下能接近全局最优（线性模型中概念之间的影响相对独立）。

**分析角度**：

- **$k_{\min}$ 的分布**：如果分布集中在低值区域（如大部分样本只需修正 1-3 个概念），说明 CBM 的干预机制非常高效——专家只需少量工作就能纠正大部分错误。
- **均值 vs 中位数**：中位数反映了"典型样本"的干预需求，均值则受极端值影响。如果中位数远小于均值，说明存在少数"难样本"需要大量干预。
- **$k=1$ 比例**：只需修正一个概念就能纠正的样本比例。这个指标在实践中很有意义——如果大量错误可以通过单一概念的修正来解决，那么自动化的概念检测 + 人工审核就是一个可行的部署方案。

### 4.4 实验 4：噪声与预算约束

**目标**：测试干预在非理想条件下的鲁棒性。

**现实考量**：前三个实验假设了一个"完美专家"——可以准确识别并修正任何概念。但在现实中：
- 专家可能给出错误的修正（噪声）
- 专家的时间和精力是有限的（预算约束）

**方法 A——噪声鲁棒性**：固定预算 $k=10$，在不同噪声水平 $\eta \in \{0, 0.05, 0.1, 0.2, 0.3, 0.5\}$ 下测试干预效果。噪声的含义是：对每个被干预的概念，以概率 $\eta$ 将修正值翻转（即本应修正为 $c^*$，实际修正为 $1 - c^*$）。

这模拟了一个"不完美的专家"——新手可能错误判断某些属性，或者标注过程中存在噪声。

**方法 B——预算 × 策略网格**：在无噪声条件下，测试预算 $k \in \{5, 10, 20\}$ × 策略 $\{\text{Random, Uncertainty, Importance}\}$ 的所有组合。这回答了一个实际的资源分配问题：给定有限的干预预算，应该选择哪种策略？

**为什么只测三种策略**：Greedy Oracle 需要知道真实标签，Error-Targeted 需要知道哪些概念预测错了——这两者在有噪声的条件下假设过强。Uncertainty 和 Importance 是部署时可用的策略，Random 作为基线，这三种足以反映现实情况。

## 6 实验流程

```bash
# 一键运行全部 4 个阶段
bash run_all.sh

# 或分步执行
python scripts/train_concept.py       # Stage 1: 训练概念预测器
python scripts/train_label.py         # Stage 2: 训练标签预测器
python scripts/run_interventions.py   # Stage 3: 执行干预实验 1-4
python scripts/visualize.py           # Stage 4: 生成 6 张可视化图
```

所有输出保存在 `outputs/` 目录下：`checkpoints/`（模型权重）、`results/`（实验数据）、`figures/`（可视化图表）。

```
cbm/
  config.py                  # 路径、超参数、设备配置
  dataset.py                 # CUB-200-2011 数据加载器（50 类、方差过滤）

  models/
    concept_predictor.py     # InceptionV3 (partial freeze) → 概念概率
    label_predictor.py       # Linear → 类别 logits（权重可解释）
scripts/
  train_concept.py           # 训练概念预测器
  train_label.py             # 训练标签预测器（基于预测概念）
  run_interventions.py       # 干预实验 1-4
  visualize.py               # 生成 6 张图表
```
